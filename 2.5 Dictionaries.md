# 🐍 The Yin Yang of Python

## 2.5 Dictionaries

---

## 🗺️ What Is a Dictionary?

A **dictionary** is a collection that stores **key-value pairs**. Every key must be paired with an assigned value.

But here is what makes dictionaries fundamentally different from everything you have seen so far. With lists and tuples, Python asks: *"In what position?"* With sets, it asks: *"Does it belong?"* With dictionaries, Python starts asking a far more interesting question:

> **"What does this mean?"**

---

## 🤔 Why Do Dictionaries Exist?

Because humans do not think only in sequences (lists, tuples) or uniqueness (sets). **We think in relationships and associations.**

The human mind works like this: name → thing. Symbol → meaning. Question → answer. Identity → attribute. **That is a dictionary.** It is the Hermetic principle made data structure: *"As above, so below"* — key and value reflecting each other.

---

## ☯️ The Yin-Yang of the Dictionary

**Yang (the Key):** Immutable (string, number, tuple). Unique (never repeats). The name that identifies. The question you ask.

**Yin (the Value):** Can be anything (mutable or immutable). Can repeat. The content you store. The answer you receive.

The perfect duality: **the stable key (Yang) sustains the dynamic value (Yin).** Like an immovable anchor that allows the ship to move with the waves.

---

## 🧾 Syntax

```python
person = {
    "name": "Yamil",
    "age": 25,
    "discipline": "Python"
}
```

Curly braces `{}`. Each entry is `key: value`. Keys must be immutable. Values can be anything.

Why must keys be immutable? **Because identity should not change.** You can change the age, the status, the attributes — but not the name of the attribute itself.

Why historically unordered? Because relationships do not have inherent order. "México → Mexico City" is not "before" or "after" "Japan → Tokyo." They are parallel associations, not a temporal sequence. (Technical note: since Python 3.7+, dictionaries preserve insertion order, but conceptually they are about **mapping, not sequence**.)

---

## ⚖️ What Problem Does a Dictionary Solve?

**1. Modeling reality.** Reality is not a list. It is a collection of properties with meaning. A person is not "element at index 0" — a person has a name, an age, a city. Dictionaries let you express that.

**2. Semantic access.** You do not search by position. You search by **meaning**. `person["name"]` reads like a question in natural language.

**3. Cognitive clarity.** Code stops being mechanical and starts being readable as human language.

## 🛠️ Dictionary Methods

### Modification methods:

**Direct assignment** — the most Pythonic way. `person["age"] = 25` adds or modifies. Clean, direct.

**`update(other_dict)`** — merges another dictionary in. Adds new keys, updates existing ones. Fusion without manual looping.

**`pop(key, default)`** — removes a key and returns its value. If the key does not exist, returns the default instead of crashing. Combines getting and eliminating in one operation.

**`popitem()`** — removes and returns an arbitrary key-value pair. For gradually emptying a dictionary.

**`clear()`** — removes all key-value pairs. Reset without creating a new dictionary.

**`setdefault(key, default)`** — returns the value of a key if it exists; if it does not, creates it with the default value. Eliminates the repetitive pattern of "check if it exists before accessing."

```python
person = {"name": "Yamil"}

# Direct assignment
person["age"] = 25

# Merge
person.update({"city": "Mérida", "discipline": "Python"})

# Pop with default
city = person.pop("city", "Not specified")  # "Mérida"

# Setdefault
score = person.setdefault("score", 0)  # Creates "score": 0
```

### Query methods (do not modify):

**`get(key, default)`** — returns the value of a key, or a default if it does not exist. **Never crashes.** The tolerant alternative to direct access (`person["key"]` raises an error if the key is missing; `person.get("key")` returns `None`).

**`keys()`** — returns a view of all keys. **`values()`** — returns a view of all values. **`items()`** — returns a view of all key-value pairs as tuples. This last one is the most complete way to iterate.

```python
person = {"name": "Yamil", "age": 25}

# Safe access
person.get("city", "N/A")  # "N/A"

# Iterate over pairs — the most common pattern
for key, value in person.items():
    print(f"{key}: {value}")
# name: Yamil
# age: 25
```

## 🗝️ Ways to Access a Dictionary

**Direct access by key:** `person["name"]` — raises `KeyError` if missing.
**Safe access with `get()`:** `person.get("name")` — returns `None` if missing.
**Check existence with `in`:** `if "name" in person` — verify before accessing.
**Iterate over keys:** `for key in person` — the default iteration.
**Iterate over values:** `for value in person.values()`.
**Iterate over pairs:** `for key, value in person.items()` — the most complete.

### When to use dictionaries:

Representing entities with properties (people, objects). Configurations (settings as key-value). Caches (fast lookup by key — O(1)). Counters (frequency of elements). Mappings (translations, conversions). Graphs (nodes → neighbors). JSON and APIs (the native format of data exchange).

### When NOT to use dictionaries:

You need guaranteed order → use a list. You only care about membership → use a set. You need immutable fixed values → use a tuple.
